## 4. Problematic samples (heuristic-fallback rows ÔÇö pick 5)

All 3 042 heuristic-fallback rows share the same defect. Mechanism: `src/data/rewriter.py::build_structured_cot` (line 104) constructs

```
<think>
<r>...</r>
<{tag}>
{body}                # <-- original teacher CoT, which already contains </think>
</{tag}>
final value matches gold; emit {answer}
</think>
```

There is no `\boxed{}` emission. The teacher's `<think>...</think>` is never stripped from `body`, so the outer `</think>` is the second close. Also: the teacher body sometimes contains `<m>` / `<c>` / `<l>` placeholder text references and uses Chinese brackets `ŃÇÉŃÇĹ` instead of `\boxed`, which is what the original `dgxchen/nemotron-cot-tong` dataset emitted.

Concrete failing examples:

1. `524cb5c6` (cryptarithm_deduce, gold `$>>\`). Ends with literal text:
   ```
   I will now return the answer in
   The answer in  is
   </think>
   </m>

   final value matches gold; emit $>>\
   </think>
   ```
   No `\boxed{}`. Two `</think>`. Reader expecting `\boxed{...}` gets nothing.

2. `0fcf912a` (cryptarithm_guess, gold `|#&!`). Same shape.

3. `f26970b3` (cipher, gold `knight studies the golden castle`). Same shape but 6 023 chars long; teacher reasoning is a sprawling search log that is mostly off-topic noise ("0 mismatches", repeated alignment scans). Likely harmful as a training target.

4. `2fb20366` (equation_numeric_deduce, gold `147`). **11 165 chars**. The body contains hundreds of trial-and-error lines ("multiplication f(24,16)=ÔÇŽ wrong / determinant ÔÇŽ=ÔÇŽ wrong / abs determinant ÔÇŽ=ÔÇŽ wrong") before finally landing on the correct rule. This is exactly the kind of behavior we *don't* want to teach a small LoRA ÔÇö long meandering search that we will pay for at inference.

5. `93481650` (equation_numeric_deduce, gold `04`). 11 135 chars. Same pathology.

These five are representative; the failure mode is identical across the 3 042 affected rows.

## 5. Structural bug deep-dive

Source: `src/data/rewriter.py` lines 92ÔÇô120.

```python
def strip_existing_boxed(cot: str) -> str:
    return re.sub(r"\\boxed\{[^}]*\}", "", cot).rstrip()

def clean_cot_body(cot: str, family: str) -> str:
    cot = strip_existing_boxed(cot)
    if family in {"gravity", "unit_conversion"}:
        cot = fix_precision_for_numeric_family(cot)
    cot = cot.strip()
    return cot

def build_structured_cot(prompt, cot, family, answer):
    ...
    body = clean_cot_body(cot, family)
    synth = f"\nfinal value matches gold; emit {answer}"
    structured = (
        f"<think>\n<r>{plan_str}</r>\n<{tag}>\n{body}\n</{tag}>\n{synth}\n</think>"
    )
    return structured
```

Three concrete bugs:
1. `clean_cot_body` strips the inner `\boxed{}` but does **not** strip the inner `<think>` / `</think>` tags from the teacher CoT, so the assembled output has two `</think>`.
2. No `\boxed{answer}` is appended after the closing `</think>`. The literal `final value matches gold; emit X` line is plain prose that the SFT loss has no reason to teach as an answer-emission protocol ÔÇö and our internal extractor (regex on `\\boxed\{...\}`) finds nothing.
3. The sub-tag chosen (`FAMILY_PLAN`) is sometimes inconsistent with what the teacher body actually contains, hence the 1 553 sub-tag-mismatch rows.

Suggested fix sketch (do NOT apply here, just for the team's notebook):
- Add `cot = re.sub(r"</?think>", "", cot)` inside `clean_cot_body`.
- Replace the `synth` line with `f"\boxed{{{answer}}}"` placed **after** the closing `</think>`, mirroring the programmatic template exactly.
- Re-run on the same source and re-QA.

## 6. Coverage gap (independent of the bug)

Even if the heuristic wrapper were fixed, the dataset has a **coverage gap**: families `equation_numeric_*` (784 rows, 10 % of the file) and `cryptarithm_guess` (154 rows) have **zero** programmatic CoTs. They depend entirely on the broken fallback. After the structural fix those rows would still carry the long, meandering, teacher-style reasoning, which is structurally valid but token-expensive (~10 k chars) and educationally noisy. The team should consider writing a programmatic solver for these two families before v0.3.0 ships.

## 7. Final verdict

**NOT FIT for training v0.3.0 as-is.** Two paths forward:

- **Option A (fastest, recommended for v0.3.0):** drop the 3 042 heuristic rows. You ship a 4 788-row dataset with 100 % structural integrity and 100 % gold-match, covering `gravity, numeral, unit_conversion, bit_manipulation, cipher, cryptarithm (12 rows)`. Two prompt families (`equation_numeric_*` and `cryptarithm_guess`) will be missing ÔÇö the LoRA will be unspecialized on ~10 % of held-out distribution if the private eval matches the training family mix.
- **Option B (cleaner, ~1 day):** fix `src/data/rewriter.py::clean_cot_body` (strip inner `</?think>`) and the closing-block assembly in `build_structured_cot` (emit `\boxed{answer}` after `</think>`). Re-run; re-QA; expect 100 % structural integrity across all 7 830 rows. The CoT length issue for `equation_numeric_*` (~10 k chars/row) will remain ÔÇö consider either truncating teacher body to N steps or writing a programmatic solver.

Either way, **do not start the 7 h training run on the current file.** The model will learn to never emit `\boxed{}` from ~40 % of its training samples, which is a regression versus v0.2.0.
